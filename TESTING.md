# TESTING.md

This repository treats testing as a release gate, not a cleanup step.

## Goal

Every completed change must provide strong evidence that:

- the logic is correct
- the CLI behavior is correct
- the packaged tool still works when installed the way a user would install it

No task is complete until that evidence exists.

## Testing Layers

Every meaningful feature should be covered at multiple layers.

### 1. Unit Tests

Use unit tests for pure logic and small isolated modules.

- ID generation
- path resolution
- config parsing
- validation logic
- template selection

These should be fast, deterministic, and exhaustive around edge cases.

### 2. Integration Tests

Use integration tests for filesystem workflows and module collaboration.

- scaffolding files into a temp directory
- reading and updating config
- validating generated structures
- matching specs and bugs across folders

These tests should exercise real file I/O in temporary directories.

### 3. End-to-End CLI Tests

Use end-to-end tests for user-facing commands.

- invoke the CLI as a real command
- pass actual arguments
- inspect exit codes, stdout, stderr, and generated files

Every user-facing command must have end-to-end coverage for its expected success path and key failure paths.

### 4. Installed-Build Verification

This is mandatory for changes that affect any user-facing command, packaging behavior, generated files, or install/update flow.

The tool must be built, installed into a dedicated verification workspace, and exercised there as if it were a freshly installed package.

This step is not optional. Source-level tests alone are not enough.

## Dedicated Verification Workspace

Keep installed-build verification rooted in `.testing/`. This is the standard location for disposable sandboxes and saved artifacts during local command verification.

Tracked structure:

```text
.testing/
├── sandboxes/
│   └── .gitkeep
└── artifacts/
    └── .gitkeep
```

### Purpose

- `.testing/sandboxes/`
  Disposable per-run workspaces created by installed-build verification tests or manual verification runs. For example, `tests/e2e/installed-init.test.ts` creates a temporary install workspace here and removes it after the run.

- `.testing/artifacts/`
  Disposable per-run artifacts such as packed tarballs, captured logs, or other debugging outputs produced during installed-build verification.

Keep both directories reproducible and disposable:

- commit only the tracked `.gitkeep` placeholders
- create per-run subdirectories beneath `.testing/sandboxes/` and `.testing/artifacts/`
- remove those subdirectories when the verification run completes unless a failure requires investigation

The verification workspace exists so an agent can always build the current version, install it into a clean sandbox, run the affected command there, and inspect the resulting files.

## Mandatory Verification Workflow

For any change that affects CLI behavior:

1. Run targeted unit and integration tests first.
2. Build the package artifact that a user would consume.
3. Create disposable subdirectories under `.testing/sandboxes/` and `.testing/artifacts/`.
4. Install that artifact into the dedicated sandbox.
5. Run every affected command in that sandbox.
6. Inspect the outputs and generated files.
7. Remove temporary verification directories when the run is complete, unless preserving failure evidence.
8. Only then treat the task as complete.

If a command was affected, the verification record must clearly show that command was actually run from the installed build.

## Definition of “Affected Command”

A command is affected if a change touches:

- the command handler
- shared logic used by that command
- config loading used by that command
- templates generated by that command
- package/build/install behavior that could change the command at runtime

When in doubt, treat the command as affected and run it.

## Release-Gate Rules

The following rules are mandatory:

- No feature is complete without at least one end-to-end success-path test.
- No command change is complete without installed-build verification.
- No packaging change is complete without fresh install verification.
- No generator/scaffolding change is complete without file-level output assertions.
- No bugfix is complete without a regression test that would fail before the fix.

## Coverage Expectations

Coverage is used as a guardrail, not as a substitute for judgment.

- Aim for full coverage of core domain logic.
- Keep command-critical paths fully covered.
- Cover error paths that a real user could trigger.
- Avoid coverage inflation through trivial assertions.

Good coverage means behavior is meaningfully asserted, not merely executed.

## Test Design Standards

- Prefer deterministic tests over mocks when filesystem behavior is under test.
- Mock only at unstable or expensive external boundaries.
- Assert observable behavior, not internal implementation details.
- Use realistic command inputs and directory layouts.
- Keep fixtures small and explicit.
- Each regression test should describe the user-visible problem it prevents.

## Agent Rules

Any agent working in this repository must treat the following as non-negotiable:

- `TESTING.md` is mandatory process.
- If you changed user-facing behavior, you must verify it through the installed-build workflow.
- If you did not run the required verification, you must explicitly say the work is not complete.
- “Tests pass” is not enough if the installed command behavior was not checked.

## What Completion Looks Like

A change is only complete when all of the following are true:

- relevant unit tests pass
- relevant integration tests pass
- relevant end-to-end tests pass
- every affected command has been exercised from an installed build
- generated files and command outputs were checked for correctness

Anything less is partial work, not finished work.
